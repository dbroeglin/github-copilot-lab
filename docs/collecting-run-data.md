# Collecting data from a Copilot CLI run

This page is the field guide for collecting the most complete record of what happened during a
GitHub Copilot CLI session. The primary artifact is Copilot's native per-session JSONL event
stream, `events.jsonl`. In the current Pier-first harness, that log is captured inside each Pier
trial under `jobs/<job>/<run-id>/<trial>/agent/copilot-session/<session-id>/events.jsonl`.

OpenTelemetry (OTel) export is a complementary live-observability path. ATIF
`trajectory.json` is a cross-agent compatibility artifact and fallback, not a replacement for
native Copilot events.

## What to collect

| Source | Keep? | Why |
| --- | --- | --- |
| `agent/copilot-session/**/events.jsonl` | Always | Primary Copilot-native timeline: turns, messages, tools, warnings, context management, tokens, and AIU cost. |
| Pier `config.json` and trial `result.json` | Always | Reconstructs task, agent, model, timing, verifier, exception, and retry/rerun context. |
| `agent/copilot-cli.jsonl` and `agent/copilot-cli.txt` | Always | Raw Copilot stdout/stderr streams. Useful when auth, startup, or CLI execution fails before native events are complete. |
| `agent/trajectory.json` | Always | ATIF trajectory emitted from Copilot events/output for cross-agent analysis and non-Copilot fallback. When local OTel is captured, assistant-step metrics also carry per-LLM-call input/output/cache-write/AIU/API-duration details. |
| `verifier/reward.txt` / `reward.json` | Always | Distinguishes "Copilot ran" from "the task was solved". |
| Pier `artifacts/` | For code tasks | Captures requested files or directories from the final sandbox state. |
| `summary.json` / `summary.md` | Derived | Human and machine summaries generated by `copilot-experiments`; rebuildable from job artifacts. |
| OTel traces/metrics | Default local file for Copilot agent runs | Standard live observability for agent, LLM, and tool spans. The harness preserves a local file by default when no explicit OTLP destination is configured, giving per-LLM-call input/output/cache-write/nano-AIU data that native session events only aggregate at shutdown. |
| Copilot `--share` markdown | Optional | Human-readable transcript. Not emitted by the Pier agent by default, but usable in standalone runs or via extra CLI args. |
| Copilot debug logs (`--log-dir`, `--log-level`) | Debug only | Useful for CLI internals, exporter startup, auth, MCP, or permissions. Treat as sensitive and bulky. |

## Publicly documented vs observed

The public GitHub docs document what Copilot CLI is and how to run it interactively or
programmatically, but they do not publish a stable schema for `events.jsonl`. The command
reference documents user-facing commands and points to help topics for `logging` and
`monitoring`. The detailed logging and OTel settings here are from `copilot --help`,
`copilot help logging`, and `copilot help monitoring` in GitHub Copilot CLI `1.0.64-0`.

Treat the event schema as observed, not contractual. Keep raw logs so old runs can be re-parsed
when Copilot changes its event fields or when the harness improves its parser.

References:

- [About GitHub Copilot CLI](https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-copilot-cli)
- [GitHub Copilot CLI command reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference)
- [OpenTelemetry OTLP exporter configuration](https://opentelemetry.io/docs/languages/sdk-configuration/otlp-exporter/)
- [OpenTelemetry GenAI semantic conventions](https://github.com/open-telemetry/semantic-conventions-genai)

## Pier-first capture path

`copilot-experiments run` now delegates execution to Pier. Pier owns sandbox creation, trial
orchestration, verifier execution, artifact transfer, and job state. The local `copilot-cli`
installed agent keeps Copilot CLI as the system under test by shelling out to the real binary.

The agent runs a command equivalent to:

```text
copilot -p <instruction> --output-format json \
        --allow-all-tools \
        --session-id <uuid> \
        --log-dir /logs/agent/copilot-session \
        [--model M] [--effort E] [--mode MODE] [--agent A] [extra args]
```

It captures:

1. combined Copilot stdout/stderr through `tee` to `/logs/agent/copilot-cli.jsonl`;
2. text output to `/logs/agent/copilot-cli.txt`;
3. native Copilot session state by copying `$HOME/.copilot/session-state/<session-id>/` to
   `/logs/agent/copilot-session/<session-id>/`;
4. local Copilot OTel file export to `/logs/agent/copilot-otel.jsonl` when no explicit OTLP
   destination is configured;
5. ATIF `trajectory.json` after the run, derived from native events when available and from raw
   Copilot JSONL otherwise, enriched with OTel per-call metrics when the OTel file is present;
6. final Pier verifier outputs and requested artifacts.

After Pier downloads trial logs, the canonical experiment-repo layout is:

```text
jobs/
  <job-name>/
    <run-id>/
      config.json
      result.json
      copilot-experiments-run.json
      summary.json          # derived by copilot-experiments
      summary.md            # derived by copilot-experiments
      <trial-name>/
        config.json
        result.json
        agent/
          copilot-cli.jsonl
          copilot-cli.txt
          trajectory.json
          copilot-otel.jsonl
          copilot-session/
            <session-id>/
              events.jsonl
        verifier/
          reward.txt
          reward.json
        artifacts/
```

`show`, `analyze`, and `inspect` read this tree. For Copilot trials they prefer native
`events.jsonl`; if no native events exist, `analyze` and summaries can fall back to
`agent/trajectory.json`.

> **Root-owned `jobs/` directories.** The Pier Docker backend runs task containers as `root`
> and bind-mounts `jobs/`, so a completed run can leave `jobs/` (and its contents) owned by
> `root`. A subsequent run started as your normal user then cannot create a new run directory
> under it and Pier fails with a bare `PermissionError` on `jobs/<job-name>`. `validate` and
> `run` preflight this and report the `jobs dir` check as not writable with the fix; to clear it
> manually, run `sudo chown -R "$(id -u):$(id -g)" jobs` (or `sudo rm -rf jobs`) and retry.

## Standalone capture recipe

For ad hoc runs outside Pier, generate a session id and copy the session state after Copilot exits:

```bash
SESSION_ID="$(uuidgen)"
RUN_DIR="runs/$SESSION_ID"
mkdir -p "$RUN_DIR" "$RUN_DIR/logs"

copilot -p "Fix the failing tests" \
  --allow-all-tools \
  --output-format json \
  --session-id "$SESSION_ID" \
  --log-dir "$RUN_DIR/logs" \
  --share "$RUN_DIR/session.md" \
  > "$RUN_DIR/stdout.txt" 2>&1

cp "$HOME/.copilot/session-state/$SESSION_ID/events.jsonl" "$RUN_DIR/events.jsonl"
```

PowerShell:

```powershell
$SessionId = [guid]::NewGuid().ToString()
$RunDir = "runs\$SessionId"
New-Item -ItemType Directory -Force -Path $RunDir, "$RunDir\logs" | Out-Null

copilot -p "Fix the failing tests" `
  --allow-all-tools `
  --output-format json `
  --session-id $SessionId `
  --log-dir "$RunDir\logs" `
  --share "$RunDir\session.md" `
  *> "$RunDir\stdout.txt"

Copy-Item "$HOME\.copilot\session-state\$SessionId\events.jsonl" "$RunDir\events.jsonl"
```

If `COPILOT_HOME` is set, Copilot stores configuration and state under that directory instead of
`~/.copilot`, so look for `$COPILOT_HOME/session-state/<session-id>/events.jsonl`.

Recommended hardening:

1. Generate and pass `--session-id`; do not discover "the newest" session after the fact.
2. Record Copilot version, command, working directory, model, effort, mode, permissions, and
   non-secret environment settings.
3. Capture stdout/stderr even if `events.jsonl` is missing.
4. Copy `events.jsonl` after the process exits, because `session.shutdown` is written at the end.
5. For code tasks, snapshot the workspace before the run and capture the final diff or artifacts.
6. Store verifier output separately from Copilot execution status.

## `events.jsonl` basics

`events.jsonl` is an append-only JSON Lines file. Each non-empty line is one JSON object. The
common top-level shape is:

```json
{
  "type": "assistant.message",
  "timestamp": "2026-01-01T00:00:01.500Z",
  "data": {}
}
```

Parsing rules:

- Use `type` as the event discriminator.
- Use `timestamp` for ordering and duration calculations.
- Treat `data` as event-specific and optional.
- Ignore unknown event types but count them.
- Expect partial logs for interrupted sessions.
- Expect resumed sessions to contain more than one `session.shutdown`; sum token/cost/request
  totals across shutdowns and use the last shutdown for final context and code-change fields.

### Sensitive content warning

Session logs can include prompts, assistant responses, repository paths, file names, tool names,
command output summaries, and code snippets. Treat raw logs as sensitive project data. Do not
publish them without review or redaction. Reasoning payloads may appear as opaque or encrypted
fields such as `reasoningOpaque` or `encryptedContent`; do not try to decode them.

## Observed native event schema

| Event type | Useful fields | What it tells you |
| --- | --- | --- |
| `session.start` | `sessionId`, `producer`, `copilotVersion`, `selectedModel`, `reasoningEffort`, `startTime`, `context.{repository,branch,cwd}` | Session identity, CLI version, initial model/effort, and workspace context. |
| `session.model_change` | `newModel` | A model switch during the session. |
| `user.message` | `content` | User prompt/message count and, when retained, prompt content. |
| `assistant.turn_start` | `turnId` | Start of an assistant turn. |
| `assistant.message` | `model`, `content`, `toolRequests[]`, `outputTokens`, `turnId` | Assistant text preview, model used for the turn, requested tools, and per-turn output tokens. |
| `tool.execution_start` | `toolCallId`, `toolName`, `model`, `turnId` | Tool invocation start; maps a tool call id to a tool name. |
| `tool.execution_complete` | `toolCallId`, `success`, `result`, `error`, `toolTelemetry.metrics` | Tool result status, output/error, latency, and result-size telemetry. Correlate by `toolCallId`. |
| `assistant.turn_end` | `turnId` | End of an assistant turn; combine with turn start timestamps for duration. |
| `session.warning` | `message` | CLI/session warnings. |
| `hook.start`, `hook.end` | Hook-specific payload | Hook execution count and timing if details are present. |
| `session.compaction_start` | `systemTokens`, `conversationTokens`, `toolDefinitionsTokens` | Context size before automatic/manual compaction. |
| `session.compaction_complete` | `preCompactionTokens`, `compactionTokensUsed.copilotUsage.{totalNanoAiu,tokenDetails[]}` | Compaction count, peak context, compaction cost, and live AIU rates by token type. |
| `session.truncation` | `preTruncationTokensInMessages`, `tokensRemovedDuringTruncation` | Context truncation count, peak message tokens, and removed tokens. |
| `session.shutdown` | `tokenDetails`, `totalNanoAiu`, `totalApiDurationMs`, `modelMetrics`, `systemTokens`, `conversationTokens`, `toolDefinitionsTokens`, `currentTokens`, `codeChanges` | Authoritative session totals for tokens, cost, requests, API time, final context composition, and code-change productivity. |

Correlation keys:

| Key | Use |
| --- | --- |
| `sessionId` | Join invocation metadata, session-state path, OTel attributes, ATIF trajectory, and Pier artifacts. |
| `turnId` | Group assistant messages and tool calls into a turn. |
| `toolCallId` | Join `assistant.message.toolRequests[]`, `tool.execution_start`, and `tool.execution_complete`. |
| `model` / `selectedModel` / `newModel` | Reconstruct the model set used by the session. |
| `timestamp` | Compute session, turn, tool, and phase durations when start/end pairs exist. |

## Token and cost accounting

`session.shutdown` is the authoritative source for session-level usage:

```json
{
  "type": "session.shutdown",
  "data": {
    "tokenDetails": {
      "input": { "tokenCount": 1000 },
      "cache_read": { "tokenCount": 8000 },
      "cache_write": { "tokenCount": 1200 },
      "output": { "tokenCount": 150 }
    },
    "totalNanoAiu": 1215000000,
    "totalApiDurationMs": 4000,
    "modelMetrics": {
      "claude-opus-4.8": {
        "requests": { "count": 2 },
        "usage": {
          "inputTokens": 10200,
          "outputTokens": 150,
          "cacheReadTokens": 8000,
          "cacheWriteTokens": 1200,
          "reasoningTokens": 40
        },
        "totalNanoAiu": 1215000000
      }
    },
    "systemTokens": 5000,
    "conversationTokens": 3000,
    "toolDefinitionsTokens": 2000,
    "currentTokens": 10000,
    "codeChanges": {
      "filesModified": ["a.py", "b.py"],
      "linesAdded": 3,
      "linesRemoved": 1
    }
  }
}
```

Collected metrics:

- token totals: non-cached input, cache reads, cache writes, output, reasoning tokens, total input,
  and grand total;
- cost: `totalNanoAiu / 1e9` gives AIU;
- per-model usage: request count, token usage, reasoning tokens, and AIU by model;
- API throughput: total API duration and request count;
- context composition: final system, conversation, tool-definition, and current-context tokens;
- productivity: files modified and lines added/removed.

`session.compaction_complete` can include live token prices:

```json
{
  "type": "session.compaction_complete",
  "data": {
    "compactionTokensUsed": {
      "copilotUsage": {
        "totalNanoAiu": 2000000,
        "tokenDetails": [
          {
            "tokenType": "input",
            "tokenCount": 10,
            "batchSize": 1000000,
            "costPerBatch": 300000000000
          }
        ]
      }
    }
  }
}
```

Use `costPerBatch / batchSize` as the per-token nano-AIU rate. The harness uses these live rates
when present and falls back to observed defaults otherwise, then normalizes the by-token-type split
to the authoritative `session.shutdown.totalNanoAiu`.

Known limitations:

- Per-turn input/cache/cost is not present in the observed native log. `assistant.message` can
  expose `outputTokens`, but input, cache, and AIU are session-level totals unless OTel export is
  enabled and retained. The harness enables local OTel file export by default for Copilot agent
  runs when no explicit OTLP destination is configured.
- Tool result size (`toolTelemetry.metrics.resultForLlmLength`) is a proxy for context injected
  back into the model, not an exact token count.
- Aborted sessions may lack `session.shutdown`; keep raw CLI streams and partial events.
- Opaque/encrypted reasoning fields are not human-readable by design.
- Event names and payload shapes can change between Copilot CLI versions.

## ATIF trajectory fallback

The Pier agent writes `agent/trajectory.json` in ATIF v1.7 format after each run. It is generated
from native Copilot events when possible, or from `agent/copilot-cli.jsonl` when native events are
missing. It captures user/agent steps, tool calls, observations, model names, and final metrics
such as prompt, completion, cached tokens, AIU, reasoning tokens, peak context, and compaction
count when those were available.

When `agent/copilot-otel.jsonl` is present, each assistant step that can be matched by Copilot
`turnId` gets ATIF `metrics.prompt_tokens`, `metrics.completion_tokens`, `llm_call_count`, and
`metrics.extra.copilot_otel`. That extra payload preserves the per-call OTel facts the standard
ATIF token fields cannot represent directly: `cache_creation_input_tokens`, `aiu`,
`server_duration_ms`, current context size, token limit, and the raw normalized `llm_calls` list.
When the exporter includes per-call cache reads, the payload also includes
`cache_read_input_tokens`. `final_metrics.extra.copilot_otel` stores the run-level OTel aggregate.
ATIF `total_cached_tokens` uses the native cache-read total when available and falls back to OTel
cache reads when OTel exposes them.

Use ATIF for cross-agent comparisons and fallback analysis. Use native `events.jsonl` for
Copilot-specific economics, compaction/truncation detail, and the most faithful timeline.

## Other Copilot CLI collection surfaces

### `--output-format json`

In non-interactive mode, `--output-format json` makes successful Copilot output JSONL on stdout.
The Pier agent stores that stream as `agent/copilot-cli.jsonl`. Keep it, but do not use it as the
primary event source when native session events exist.

### `--share`

`--share=<path>` writes a markdown transcript after a non-interactive session completes. It is
useful for ad hoc human review. The Pier agent does not enable it by default because Pier already
captures stdout, ATIF, native events, verifier outputs, and requested artifacts.

### `--log-dir` and `--log-level`

`--log-dir <directory>` writes Copilot CLI diagnostic logs. The default location is
`~/.copilot/logs/`. In the Pier agent, `--log-dir` points at `/logs/agent/copilot-session`, while
the canonical native event log is still copied from `$HOME/.copilot/session-state/<session-id>/`.

`--log-level` accepts:

| Level | Meaning |
| --- | --- |
| `none` | No logging output. |
| `error` | Errors only. |
| `warning` | Errors and warnings. |
| `info` / `default` | Errors, warnings, and info. |
| `debug` / `all` | Full debug logging. |

Use debug logs only when troubleshooting CLI internals, exporter startup, auth, MCP setup, or
permission issues. Do not retain them as routine experiment artifacts without a redaction policy.

## OpenTelemetry export

Copilot CLI can export traces and metrics through OpenTelemetry. OTel is off by default and
activates when any of these is true:

- `COPILOT_OTEL_ENABLED=true`
- `OTEL_EXPORTER_OTLP_ENDPOINT` is set
- `COPILOT_OTEL_FILE_EXPORTER_PATH` is set

### Exporter options

| Variable | Meaning |
| --- | --- |
| `COPILOT_OTEL_ENABLED` | Explicitly enable OTel; defaults to `false`. |
| `COPILOT_OTEL_EXPORTER_TYPE` | `otlp-http` (default) or `file`; auto-selects `file` when `COPILOT_OTEL_FILE_EXPORTER_PATH` is set. |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | OTLP endpoint URL; setting it auto-enables OTel. For OTLP/HTTP, the common local collector endpoint is `http://localhost:4318`. |
| `OTEL_EXPORTER_OTLP_PROTOCOL` | For `otlp-http`: `http/json` (Copilot default) or `http/protobuf`. Per-signal overrides: `OTEL_EXPORTER_OTLP_TRACES_PROTOCOL`, `OTEL_EXPORTER_OTLP_METRICS_PROTOCOL`. |
| `COPILOT_OTEL_FILE_EXPORTER_PATH` | Writes all signals to one JSONL file; setting it auto-enables OTel. |
| `COPILOT_OTEL_SOURCE_NAME` | Instrumentation scope name; defaults to `github.copilot`. |
| `OTEL_SERVICE_NAME` | Service name resource attribute; defaults to `github-copilot`. |
| `OTEL_RESOURCE_ATTRIBUTES` | Extra comma-separated resource attributes, useful for job names, trial names, task slugs, variants, and session ids. |
| `OTEL_EXPORTER_OTLP_HEADERS` | Headers for the OTLP exporter, for example `Authorization=Bearer ...`; prefer secret managers or env files. |
| `OTEL_EXPORTER_OTLP_CERTIFICATE` | Extra CA bundle for HTTPS OTLP endpoints; per-signal trace/metric variants are supported. |
| `OTEL_EXPORTER_OTLP_CLIENT_CERTIFICATE` / `OTEL_EXPORTER_OTLP_CLIENT_KEY` | mTLS client certificate and unencrypted key; both must be set together and only apply to HTTPS endpoints. |
| `OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT` | Include full prompt/response messages, tool arguments/results, system instructions, and tool definitions; defaults to `false`. |
| `OTEL_LOG_LEVEL` | OTel diagnostic log level: `NONE`, `ERROR`, `WARN`, `INFO`, `DEBUG`, `VERBOSE`, or `ALL`. |

Copilot CLI's current OTel help says `grpc` is not supported for its `otlp-http` exporter path; an
unsupported protocol warns and falls back.

### File exporter in Pier

The Pier `copilot-cli` agent now enables local file export by default when neither
`COPILOT_OTEL_FILE_EXPORTER_PATH` nor `OTEL_EXPORTER_OTLP_ENDPOINT` is already configured. The
default file is `/logs/agent/copilot-otel.jsonl`; Pier downloads it with the rest of `agent/` logs.
The agent also adds resource attributes for the generated Copilot session id and agent name, and
folds matching `chat <model>` spans into `agent/trajectory.json` step metrics after the run.

To override the default, set OTel environment variables on the Pier agent so Copilot inherits them:

```yaml
agents:
  - name: copilot-cli
    model_name: gpt-5-mini
    env:
      COPILOT_OTEL_FILE_EXPORTER_PATH: /logs/agent/copilot-otel.jsonl
      COPILOT_OTEL_SOURCE_NAME: copilot-experiments
      OTEL_SERVICE_NAME: copilot-experiments
      OTEL_RESOURCE_ATTRIBUTES: experiment.job=example-fix-bug,agent=copilot-cli
    kwargs:
      reasoning_effort: low
```

If `OTEL_EXPORTER_OTLP_ENDPOINT` is set, the harness does not force a local file path; set
`COPILOT_OTEL_FILE_EXPORTER_PATH` explicitly if you want the local file in addition to custom OTel
configuration.

For standalone local capture:

```bash
COPILOT_OTEL_FILE_EXPORTER_PATH="$RUN_DIR/copilot-otel.jsonl" \
OTEL_SERVICE_NAME="copilot-experiments" \
OTEL_RESOURCE_ATTRIBUTES="session.id=$SESSION_ID,experiment=ad-hoc" \
  copilot -p "Fix the failing tests" \
    --allow-all-tools \
    --output-format json \
    --session-id "$SESSION_ID"
```

The file exporter writes JSON Lines. In Copilot CLI `1.0.64-0`, each line was a simplified record
with `type: "span"` or `type: "metric"` rather than a raw OTLP envelope. Span records include
`traceId`, `spanId`, optional `parentSpanId`, `name`, `kind`, `startTime`, `endTime`,
`attributes`, `events`, `status`, `resource`, and `instrumentationScope`. Resource attributes
include `OTEL_RESOURCE_ATTRIBUTES`, `OTEL_SERVICE_NAME` as `service.name`, and Copilot's
`service.version`. Metric records include `name`, `description`, `unit`, and `dataPoints`.

Metric records can appear more than once in the file because metrics are cumulative export
snapshots. For per-request analysis, use spans as the source of truth and treat metrics as
aggregate observability signals.

### OTLP/HTTP exporter

Use OTLP/HTTP to send telemetry to an OTel collector or SaaS backend:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318 \
OTEL_RESOURCE_ATTRIBUTES="experiment.job=tracer-bullet,agent=copilot-cli" \
  copilot -p "Fix the failing tests" --allow-all-tools
```

With protobuf:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318 \
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf \
  copilot -p "Fix the failing tests" --allow-all-tools
```

With authentication and content capture:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=https://collector.example.com:4318 \
OTEL_EXPORTER_OTLP_HEADERS="Authorization=Bearer $TOKEN" \
OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT=true \
  copilot -p "Fix the failing tests" --allow-all-tools
```

Only enable content capture in trusted environments. It can include prompts, source code, file
contents, tool arguments, tool results, and system/tool instructions.

### What Copilot exports through OTel

From `copilot help monitoring`, Copilot emits traces for agent interactions, LLM calls, and tool
executions. The typical trace shape is:

```text
invoke_agent
  plan
    chat <model>
    execute_tool <tool>
  chat <model>
  execute_tool <tool>
```

Spans carry attributes such as model name, token counts, durations, costs, and error information.
Subagent invocations are linked into the same trace through context propagation.

Observed span records from local file-exporter probes:

| Span | Useful fields | What it adds beyond `events.jsonl` |
| --- | --- | --- |
| `invoke_agent` | `gen_ai.conversation.id`, `gen_ai.agent.id`, `gen_ai.agent.version`, `gen_ai.request.model`, `github.copilot.agent.type`, `github.copilot.turn_count`, aggregate `gen_ai.usage.*`, aggregate `github.copilot.nano_aiu`, `github.copilot.context.skills`, `github.copilot.context.mcp_server_names`, `github.copilot.context.custom_agent_names` | Agent-level trace root and aggregate usage/cost for the invocation. MCP server names are hashed. |
| `chat <model>` | `gen_ai.conversation.id`, `github.copilot.turn_id`, `gen_ai.request.model`, `gen_ai.response.model`, `gen_ai.response.id`, `gen_ai.response.finish_reasons`, `gen_ai.usage.input_tokens`, optional cache-read usage fields, `gen_ai.usage.cache_creation_input_tokens`, `gen_ai.usage.output_tokens`, `github.copilot.nano_aiu`, `github.copilot.server_duration`, `github.copilot.interaction_id`, `github.copilot.service_request_id` | Per-LLM-call input/output/cache-read/cache-write tokens when exported, nano-AIU, and server duration. This is the main data source for turn-level economics. |
| `execute_tool <tool>` | `gen_ai.operation.name`, `gen_ai.tool.name`, `gen_ai.tool.call.id`, `gen_ai.tool.type`, `gen_ai.provider.name`, span status | Standard tool spans that can be joined to chat spans by trace and to native events by tool-call id. Native `tool.execution_complete.toolTelemetry` remains richer for result sizes and file/code metadata. |

`chat <model>` spans also emitted a `github.copilot.session.usage_info` event with
`github.copilot.current_tokens`, `github.copilot.token_limit`, and
`github.copilot.messages_length`. Hook execution appeared as `github.copilot.hook.start` /
`github.copilot.hook.end` span events. The `invoke_agent` span emitted MCP lifecycle events and a
`github.copilot.user.message` event.

Per-call token/cost interpretation:

| Field | Interpretation |
| --- | --- |
| `gen_ai.usage.input_tokens` | Total prompt/input tokens for that LLM request. In observed runs, summing chat spans matched `session.shutdown.modelMetrics.<model>.usage.inputTokens`. |
| `gen_ai.usage.cache_read_input_tokens` | Cache-read tokens for that request when exported. The parser also accepts common aliases such as `gen_ai.usage.cached_tokens`. |
| `gen_ai.usage.cache_creation_input_tokens` | Cache-write tokens for that request. In observed runs, summing chat spans matched `session.shutdown.tokenDetails.cache_write.tokenCount`. |
| `gen_ai.usage.output_tokens` | Output tokens for that request. |
| `github.copilot.nano_aiu` | Per-request cost in nano-AIU. Prefer this over `github.copilot.cost` for economics; summing observed chat spans matched `session.shutdown.totalNanoAiu`. |
| `github.copilot.server_duration` | Per-request server/API duration in milliseconds; summing observed chat spans matched `session.shutdown.totalApiDurationMs`. |
| `github.copilot.turn_id` | Join key to native `assistant.turn_*` / `assistant.message` events. |

Some observed OTel spans did **not** split per-call `input_tokens` into non-cached input versus
cache-read tokens. When the exporter provides cache-read fields, the parser captures them; otherwise
that split remains session-level in `session.shutdown.tokenDetails`.

Metrics currently documented by the CLI help:

| Metric | Meaning |
| --- | --- |
| `gen_ai.client.operation.duration` | LLM/agent call duration histogram, in seconds. |
| `gen_ai.client.token.usage` | Token usage histogram by type. |
| `github.copilot.tool.call.count` | Tool invocation counter. |
| `github.copilot.tool.call.duration` | Tool execution latency histogram, in seconds. |
| `github.copilot.agent.turn.count` | LLM round-trips per invocation histogram. |

Local probes also observed `github.copilot.mcp.server.connection.count`, a counter of completed MCP
server connection attempts by transport and outcome.

OTel is best for live, cross-session observability and standard backend integrations.
`events.jsonl` remains the richer per-session forensic artifact, but OTel is the only observed
surface that currently exposes per-LLM-call input/cache-write/AIU fields. `analyze` auto-discovers
`agent/copilot-otel.jsonl` for Pier jobs; for direct file analysis, pass it explicitly:

```bash
uv run copilot-experiments analyze --file events.jsonl --otel-file copilot-otel.jsonl
```

## Collection checklist

1. Define Pier task directories and a Pier `JobConfig`.
2. Run Copilot through the `copilot-cli` installed agent with explicit model/effort/mode.
3. Let the agent generate a session id and copy native session state into `agent/copilot-session/`.
4. Keep `agent/copilot-cli.jsonl` and `.txt` for diagnostics.
5. Keep `agent/trajectory.json` for ATIF and fallback analysis.
6. Keep verifier rewards and requested artifacts.
7. Generate summaries from the job tree.
8. Keep the default `agent/copilot-otel.jsonl`, or configure an OTLP collector / explicit file path
   and add Pier identifiers through `OTEL_RESOURCE_ATTRIBUTES`.
9. Redact secrets and review raw logs before sharing any artifacts.
